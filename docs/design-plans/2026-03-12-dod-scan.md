# DOD Contract Scanner Design

## Summary

`dod-scan` is a Python CLI data pipeline that scrapes the U.S. Department of Defense's daily contract award announcements from war.gov, extracts structured data from the semi-standardised HTML, and uses an LLM to filter out service contracts — leaving only procurement contracts (physical goods / hardware). The surviving contracts are then geocoded to lat/lon coordinates and exported as KML files for tools like Google Earth, with an optional interactive Mapbox dashboard.

The pipeline is built as five discrete stages (scrape, parse, classify, geocode, export) that communicate exclusively through a shared SQLite database. This means any stage can be re-run independently after a failure without losing work from the stages that already completed. The whole thing is designed to run unattended via a daily cron job, with idempotent stage logic ensuring re-runs on the same day don't duplicate work.

## Definition of Done
A Python CLI tool (`dod-scan`) that runs daily via cron to scrape DOD contract postings from war.gov, parse contract details, classify them as procurement vs service using an LLM (OpenRouter or Claude API), geocode work locations (falling back to company HQ when work location is unspecified), and export filtered procurement contracts as KML files and optionally as a Mapbox HTML dashboard. Includes a README with setup instructions for both backfill and daily cron scheduling. Logs errors to files.

## Acceptance Criteria

### dod-scan.AC1: Scraping daily contract pages
- **dod-scan.AC1.1 Success:** Scraper fetches today's contract page and stores raw HTML in SQLite
- **dod-scan.AC1.2 Success:** `--backfill N` paginates through N pages of historical listings
- **dod-scan.AC1.3 Success:** Already-scraped pages (by article ID) are skipped without re-fetching
- **dod-scan.AC1.4 Failure:** 403 from httpx triggers Playwright fallback transparently
- **dod-scan.AC1.5 Failure:** Network errors logged to file, scrape stage exits non-zero

### dod-scan.AC2: Parsing contract entries
- **dod-scan.AC2.1 Success:** Each contract paragraph extracted as a separate row with branch, company name, company city/state, dollar amount, contract number, work locations, completion date, contracting activity
- **dod-scan.AC2.2 Success:** Modification codes (e.g. P00045) extracted and `is_modification` set correctly
- **dod-scan.AC2.3 Success:** Multiple work locations with percentages parsed into JSON array
- **dod-scan.AC2.4 Success:** Raw text preserved verbatim for every contract entry
- **dod-scan.AC2.5 Edge:** Small business asterisk (*) stripped from company name
- **dod-scan.AC2.6 Edge:** "Work locations and funding will be determined with each order" results in empty work_locations (triggers HQ fallback in geocoding)

### dod-scan.AC3: LLM classification
- **dod-scan.AC3.1 Success:** Unclassified contracts sent to configured LLM and result stored with classification, confidence, reasoning, model
- **dod-scan.AC3.2 Success:** Already-classified contracts skipped on re-run
- **dod-scan.AC3.3 Success:** OpenRouter provider works with `LLM_PROVIDER=openrouter`
- **dod-scan.AC3.4 Success:** Anthropic provider works with `LLM_PROVIDER=anthropic`
- **dod-scan.AC3.5 Failure:** LLM API error logged to file, classify stage exits non-zero
- **dod-scan.AC3.6 Edge:** Malformed LLM JSON response handled gracefully (logged, contract marked for retry)

### dod-scan.AC4: Geocoding locations
- **dod-scan.AC4.1 Success:** Work location geocoded when present in contract text
- **dod-scan.AC4.2 Success:** Company HQ location used as fallback when work location is unspecified or "TBD"
- **dod-scan.AC4.3 Success:** Cached locations returned without API call
- **dod-scan.AC4.4 Failure:** Geocoding API failure logged, contract skipped (doesn't block pipeline)
- **dod-scan.AC4.5 Edge:** Multiple work locations — primary location (highest percentage or first listed) used for pin

### dod-scan.AC5: KML export
- **dod-scan.AC5.1 Success:** Valid KML file generated with one placemark per geocoded procurement contract
- **dod-scan.AC5.2 Success:** Placemarks coloured by dollar amount (green -> yellow -> red gradient)
- **dod-scan.AC5.3 Success:** Placemark popup contains company name, dollar amount, contract number, branch, description, completion date
- **dod-scan.AC5.4 Success:** `--since DATE` filters to contracts from that date onward
- **dod-scan.AC5.5 Success:** `--branch ARMY` filters to specified branch only

### dod-scan.AC6: Mapbox dashboard export
- **dod-scan.AC6.1 Success:** Self-contained HTML file generated with Mapbox GL JS map when MAPBOX_TOKEN is set
- **dod-scan.AC6.2 Success:** Clicking a pin shows popup with full contract details
- **dod-scan.AC6.3 Success:** Sidebar/panel filters by date range, branch, and dollar amount
- **dod-scan.AC6.4 Failure:** No MAPBOX_TOKEN + `--format all` produces KML only with log message
- **dod-scan.AC6.5 Failure:** No MAPBOX_TOKEN + `--format map` exits with helpful error message

### dod-scan.AC7: Orchestration and documentation
- **dod-scan.AC7.1 Success:** `run-all` executes scrape -> parse -> classify -> geocode -> export in sequence
- **dod-scan.AC7.2 Failure:** `run-all` stops on first stage failure with non-zero exit code
- **dod-scan.AC7.3 Success:** All stages log to file (configurable path)
- **dod-scan.AC7.4 Success:** README contains venv setup, dependency install, env var configuration, backfill command, daily cron entry, and output file locations
- **dod-scan.AC7.5 Success:** A new user can follow README from zero to running scheduled scans

## Glossary

- **war.gov**: The U.S. Department of Defense public-facing website, which publishes a daily listing of awarded contracts at `/News/Contracts/`.
- **KML (Keyhole Markup Language)**: An XML-based format for representing geographic data (points, paths, polygons). Readable by Google Earth, Google Maps, and GIS tools.
- **Placemark**: A KML element representing a single pin on a map, optionally with a popup description and icon style.
- **Mapbox GL JS**: A JavaScript library for rendering interactive, GPU-accelerated maps in a browser using the Mapbox tile service.
- **OpenRouter**: A third-party API aggregator that provides a single OpenAI-compatible endpoint for routing requests to many underlying LLM providers.
- **Anthropic SDK**: The official Python client library for calling Claude models directly via Anthropic's API.
- **Nominatim**: A free geocoding API run by OpenStreetMap that converts place names and addresses into latitude/longitude coordinates. Subject to a 1 req/s rate limit.
- **Census Bureau geocoding**: A free U.S. government geocoding API, used as an alternative to Nominatim for resolving U.S. addresses.
- **httpx**: A modern Python HTTP client with async support, used here as the primary transport for fetching web pages.
- **Playwright**: A browser automation library that can drive a real headless browser — used as fallback when war.gov returns a 403 to plain HTTP requests.
- **BeautifulSoup**: A Python library for parsing and navigating HTML/XML documents.
- **simplekml**: A Python library for constructing KML files programmatically.
- **typer / click**: Python libraries for building CLI applications with subcommands, argument parsing, and help text generation.
- **pydantic-settings**: A Pydantic extension for loading and validating configuration from environment variables and `.env` files.
- **pyproject.toml**: The modern Python project metadata file (PEP 517/518) used to define package dependencies, entry points, and build configuration.
- **src/ layout**: A Python packaging convention where source code lives under `src/package_name/` rather than at the repo root, preventing accidental imports of uninstalled code during development.
- **Idempotent**: Describes an operation that produces the same result if run multiple times — here, re-running a stage skips already-processed records rather than duplicating them.
- **LLM provider abstraction**: A design pattern where both LLM backends (Anthropic and OpenRouter) implement the same interface, letting the pipeline switch providers via config without changing pipeline logic.
- **Modification code**: A code in DOD contract announcements (e.g. `P00045`) indicating the record is a modification to an existing contract rather than a new award.
- **Contracting activity**: The specific DOD office or command that administered and awarded the contract (e.g. "Army Contracting Command — Aberdeen Proving Ground").
- **Procurement vs service classification**: The binary distinction the LLM is asked to make — "procurement" means the government is buying physical goods or equipment; "service" means it's buying labour, maintenance, or other non-tangible work.

## Architecture

Stage-based pipeline with five CLI subcommands sharing SQLite as the coordination layer. Each stage reads from and writes to the database, enabling independent re-runs on failure without losing prior work.

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  SCRAPE  │───▶│  PARSE   │───▶│ CLASSIFY │───▶│ GEOCODE  │───▶│  EXPORT  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
     │               │               │               │               │
     ▼               ▼               ▼               ▼               ▼
   SQLite          SQLite          SQLite          SQLite        KML + HTML
  (raw HTML)    (parsed entries)  (filtered)    (with coords)   (Mapbox map)
```

**Scrape** fetches daily contract pages from `war.gov/News/Contracts/`. Primary transport is httpx with realistic browser headers; falls back to Playwright headless browser on 403. Stores raw HTML in SQLite. Deduplicates by article ID. Supports `--backfill N` for historical pagination.

**Parse** extracts individual contract entries from stored HTML using regex. Contracts are grouped by branch headers (`AIR FORCE`, `ARMY`, `NAVY`, `DEFENSE LOGISTICS AGENCY`, `MISSILE DEFENSE AGENCY`) in centered bold `<p>` tags. Each contract is a single `<p>` tag with standardized language. Extracted fields: company name/location, dollar amount, contract number, modification code, work locations, branch, completion date, contracting activity.

**Classify** sends each unclassified contract to an LLM (configurable: OpenRouter or Claude API) to determine procurement vs service. Stores classification, confidence score, reasoning, and model used. Only procurement items pass to subsequent stages.

**Geocode** resolves locations to lat/lon coordinates. Primary: work location extracted from contract text ("Work will be performed at..."). Fallback: company headquarters location when work location is "TBD" or unspecified. Results cached in SQLite — repeated locations (Arlington VA, St. Louis MO) geocoded once. Uses free geocoding (Census Bureau or Nominatim).

**Export** generates KML files (always) and a Mapbox GL JS HTML dashboard (when `MAPBOX_TOKEN` is configured). No token = KML only with a log message. Pins placed at geocoded locations, coloured by dollar amount. Supports filtering by date range, branch, and dollar amount.

A `run-all` command chains all five stages sequentially, stopping on first failure.

### LLM Provider Interface

Two providers share a common interface:

- **Anthropic**: Direct Claude API via `anthropic` Python SDK
- **OpenRouter**: OpenAI-compatible chat completions at `https://openrouter.ai/api/v1`

Configuration selects provider via `LLM_PROVIDER` env var. Both accept the same classification prompt and return structured JSON.

### SQLite Schema

```sql
pages (
    article_id       TEXT PRIMARY KEY,
    url              TEXT NOT NULL,
    publish_date     DATE,
    scraped_at       TIMESTAMP,
    raw_html         TEXT
)

contracts (
    id               INTEGER PRIMARY KEY,
    article_id       TEXT REFERENCES pages,
    branch           TEXT,
    company_name     TEXT,
    company_city     TEXT,
    company_state    TEXT,
    dollar_amount    REAL,
    contract_number  TEXT,
    mod_code         TEXT,
    is_modification  BOOLEAN,
    work_locations   TEXT,        -- JSON array of {city, state, pct}
    completion_date  TEXT,
    contracting_activity TEXT,
    raw_text         TEXT,
    parsed_at        TIMESTAMP
)

classifications (
    contract_id      INTEGER PRIMARY KEY REFERENCES contracts,
    is_procurement   BOOLEAN,
    confidence       REAL,
    reasoning        TEXT,
    model_used       TEXT,
    classified_at    TIMESTAMP
)

geocode_cache (
    location_key     TEXT PRIMARY KEY,
    latitude         REAL,
    longitude        REAL,
    geocoded_at      TIMESTAMP
)
```

### Configuration

Environment variables (`.env` file):

```
LLM_PROVIDER=openrouter
LLM_API_KEY=sk-...
LLM_MODEL=anthropic/claude-haiku-4-5-20251001
MAPBOX_TOKEN=pk-...              # optional
DATABASE_PATH=./dod_scan.db
OUTPUT_DIR=./output
```

## Existing Patterns

This is a greenfield project — no existing codebase patterns to follow. The design introduces:

- Stage-based CLI pipeline using `typer` or `click` for subcommands
- SQLite as coordination layer between pipeline stages
- `src/` layout with `pyproject.toml` for packaging

These are standard Python patterns for CLI data pipeline tools.

## Additional Considerations

**Bot protection:** war.gov currently returns 403 on plain HTTP requests. The Playwright fallback handles this, but the site could add further protections (CAPTCHAs, rate limiting). If this happens, the scraper stage will fail and log the error — manual investigation would be needed.

**Geocoding rate limits:** Nominatim has a 1 request/second policy. The geocode cache mitigates this for steady-state use, but a large backfill could hit rate limits. The geocoder should respect this with appropriate delays.

**LLM classification edge cases:** Some contracts are genuinely ambiguous (e.g., "construction of a towboat" — is it construction or procurement of a vehicle?). The confidence score and reasoning fields allow auditing and prompt tuning over time.
